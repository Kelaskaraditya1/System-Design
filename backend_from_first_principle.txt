<--------------------------------------------------------------------------------->Lecture 2: Http Protocol<---------------------------------------------------------------------------------------------------->

1) Http protocol are stateless means it donot have memory of itself , we have to provide all the necessary information to the server so that it responds back properly and 
after than it forgets about the interaction.
than again for another request again we have to give all the necessary information like headers, tokens etc. this is stateless behaviour.

2) Http is a protocol which uses TCP(Transmission Control protocol) for establishing connection between client and server

3) Idempotent vs Non Idempotent:

it simply means that how much the state of the server changes on the request when we hit the same request once vs again and again.

Idempotent request: GET, PUT, DELETE

GET: hit once or again gives same result again and again
DELETE: first time data is deleted but than for next request it gives the same result that is 404 (NOT FOUND).
PUT: since the request is same so the data will also be same so it updates the same data again and again so no change in state.

Non Idempotent request: POST

POST: creates new resource again and again therefore non Idempotent.

Options Method/CORS(Cross Origin Request)

Cors Origin has 2 types of request:

1) Normal request: when the client accesses some another domain and server gives response than it adds a Access-Control-Allow_Origin it contains the Client's domain , if it is present than the client
will parse the response and if it is absent than it will throw CORS error.

2) Preflight request: It is a request which is raised before the actual request , just to verify whether the server allows the origin, method, header etc.
for a request to be Preflight request it has to be Cross origin request and either of the below condition has to be satisfied:
    1) the method should not be GET, POST it could be PUT or DELETE.
    2) it should have non simple headers like Authorization X-custom-headers
    3) the content/type should be other than application/x-www-form-urlencoded multipart/form-data and text/plain that is it should be application/json.

since we mostly add Authorization header also the content type of the request is content/Json therefor most of the request are Preflight request.

this request returns a 204(Ok + No response body) and it is just an enquiry request that does the server allows the client's origin and the methods and headers that the client is going to use.

Status Codes:

1) 1xxx: Informational 
    commonly used, when massive size request in this case the client sends first headers and it returns some code on the basis of that it sends the request-body
                    when switching protocols like switching from Http to Websocket.

2) 2xxx: Succeess
    200 (Ok) , 201(Ok + Creation of resource), 204(Ok + No response body).

3) 3xxx: Redirection
    301(Permenant Redirection) resource has been moved to another route permenantly.
    302(Temporary Redirection)

4) 4xxx: client side error
    400(Bad request), 401(UnAuthorized), 403(Forbiden), 404(Not found), 405(Method not allowed), 409(Conflict), 429(Rate Limit)

5) 5xxx: Server side error
    500(Interval server error), 503(Service unavailable), 504(Server Timeout)

Content Negotiation: this is a concept in which the client and server agrees on the best format to exchange data. could be Json,Xml or Html.
there are 3 types of Content Negotiation
    1) Media Negotiation: the client specifies the desired format through Accept headers , for eg: content-type/json
    2) Language Negotiation: client requests the content in a particular language and it uses Accept language header
    3) Encoding Negotiation: client specifies which encoding format it supports.

<--------------------------------------------------------------------------------->Lecture 4: Authentication & Authorization<---------------------------------------------------------------------------------------------------->

Http was stateless means it donot have memory of itself , but we require stateful interaction for that session came into picture.

1) Session:

Session gives server temprory memory/context of the user, whether the user is authenticated or not , what items has user has saved in the cart etc along with session id.

whenever client first time interacts with server the server creates a session id and creates session for user and sends this session id to the client 
which is saved in browser cookies by the client/frontend and server stores both session id and data.
Client/Browser saves this session id in Cookies.
So next time Client makes the request it attaches the cookies and from cookies it get's the session id for accessing the stored data from the session. 

Session used to store information in file system than in database than in cache memory.

Disadvantage of session is that when application scales it gives difficulty to share data accross multiple servers 

2) Json Web Token(Jwt):

jwt token contains of 3 parts header.payload.signature.
header contains the information of which encryption algorithm is used like Hmac Sha256.
payload contains crucial information such subject which can be userId than  issued at and expired at 
than there is signature which the result of signing of Base 64 url encoding combination of header and payload and secret key.

the secret key is shared accross the servers so that every server can validate the jwt token.

while validating it takes the header and payload reruns the base64 encoding gets the data and using secret key calculates the signature that is the 3rd part and compares it with the original Jwt Token.

Because of Jwt Authentication becomes Stateless instead of saving auth status in session.

Disadvantage: since it is stateless the server has no memory that the user is authenticated or not so if the jwt token is stolen and the hacker sends a request it will consider the request to be authenticated.


3) Cookies:

cookies is a mechanism a key value pair which the server stores on the client browser. and the server has the access to the cookie which is stored on the clients browser.
cookie given by one server cannot be accessed by another server.

workflow: 
first time user authenticates using username and password to the server.
In return server stores the cookie which contains jwt/session id in the clients browser.
than in every subsequent request the client sends cookies which can be accessed by the server for validating the authenticity of the user.

The Backend structure is broken brodly in 3 stages:
1) Controller: for reciving request and giving back response to the client
2) Service: where it executes buisness logic
3) Repository: layer which interacts with database or external api's.

so the incoming request has to be in a specific format, like it should have all the necessary fields and the fields should have proper values as per requirements.
so this validation of incoming request can be done with a middleware which intercepts and checks the request body before it reaches the controller.

Types of validation:

1) Syntactic validation: checking whether proper type of data is provided or not , like email should contain @gmail.com , phone number should contain country code.
2) Symantic validation: checking whether data makes sense or not, for eg the age>0 && age<120 than the birth date cannot be in current date or future date etc. 
3) Type validation: checking the expected(what type is required) and actual type of data which is submitted.

 <--------------------------------------------------------------------------------->Lecture 6: Background Tasks <---------------------------------------------------------------------------------------------------->

Background Tasks are tasks which run their execution our of the normal request response cycle , asynchronously without depending or making some service dependent on it self.

for eg: There is a signup process in which when we enter the details and send a verification email to the user , and now there is another service which handels the sending of the email.
and it might not be as responsive like it might go down , so because of this the entire process will fail since it is a synchronous request if there is no proper error handeling 

so the solution for this might be make a queue and a background process , once the user signups it will serialize the data and send it to the queue and than the email service will 
take data from the queue deserialize and use it for sending email. so by this we made the process asynchronous ,so that even if the process fails it fails gracefully instead of failing 
the entire process.
So basically background tasks provide asynchronous behaviour instead of waiting for a job to completed , we suspend it and move the execution ahead.

Task Queue/Broker:

This follows a Pub/Sub model.
The publisher creates a new task and pushes into the queue and the consumer which has a background process constantly monitors the queue for the incoming message and executes them 
outside of the req/res cycle.
after sucessful completion of the task the consumer sends a acknowledgement to the queue and than it can decide whether to remove the task from the queue or not.

Design consideration when working with task queue:

1) Idempotency: the task should be designed in such a way , even if it fails it should not cause any side effect.
it should be a complete single transaction means if failed it should be roll backed to the previous state sucessfully and next time the transaction should start from scratch.  

2) Error handeling: should impement proper logging, retry mechanism 

3) Monitoring: at any instance we should have a complete picture about the pending tasks in the queue no of sucessful task , nno of failed task reason behind the failed tasks.
for this we should use proper monitoring tools such as Promethus/Grafana and add proper metric for clearity.

4) Scaling: add proper horizontal or vertical scaling mechanism as per the traffic etc.

5) Rate Limiting: add proper rate limiting mechanism so that if there it is making request to external services it should not overload by sending multiple requests to it.

<--------------------------------------------------------------------------------->Lecture 8: Scaling & Performance<---------------------------------------------------------------------------------------------------->

 Latency: the time taken for the request to complete it's cycle or give the error is known as Latency.
 in simple terms , the time taken by the client to raise the request, it reaches the server, the server responses and client processess that response is called as Latency.

 Throughput: how many requests the system can handle in a given unit of time.

Utilization: the amount of use of the entire system at a given instance of time. 

Bottle neck: the component or that part of the system which makes it slower.
Important note: whenever the latency increases measure each component indivisually and find the real bottle neck component, instead of playing the guessing game. 

Percentile math with latency:

There are 3 figures which we have to consider while trying to figure the latency of the system.
1) P50 which is how much 50% of our users are facing latency.
2) p90 which is how much 10% of our users are facing latency.
3) p99 which is how much 1% of our users are facing latency.

and p99 is top than comes p90 and p50.
this figures help us understand latency of the system.

Performance related issues which are related to Database:
1) N+1 Query problem (1 + N extra query):

We have to show post realted information on our frontend , where above there is a multimedia file and below is information of the author.
and our api only returns image of the post , but it does not returns the information about the user.
than for every post id we fetch the information of the author , so in total '1' query for fetching all the posts and rest 'n' requests for fetching the author information.

which could have been done in a single request or 2 requests.
 like in a single request all information, or in a single request get all the posts and than take an array of user_id and get the list of information of users.
 so the task which requires 'n+1' requests can be done in single request.

2) Lack of indexes: we have to monitor our system and than decide that some computing are happening again and again so we can add index to that columns.

3) Database Connections: Locally database connections are cheap but when the app scales but it creates issues when the app starts to scale.
also it requires a lot of resources to database connection and the number of connections get exhausted quickly, so to solve this problem Connection pooling was introduced.

Connection Pool: it is a pool which has a bunch of open connections and when the request arrives it goes to the connection pool where it has open connection which serves the request 
and than the open connection is given to the pool again which can be reused again to another request.

types of connection pool:

2) External: in external connection we use a external service to manage the connections pool for database connections. for eg: Pg bouncer.   


Cacheing patterns: 

1) Read aside: (which we usually use)
whenever we want to find anything, first we check in the cache whether it exists or not , if not we find it in cache and than store in the cache and than returns to the request.
and when we update that resource than the stale data should be invalidated and get the fresh data from the db.

2) Write through: 
when there is a update operation , the system updates the db and cache simultaniously.

3) Write behind:(Clever approach with risk)

In this approach whenever there is a update operation we update the data in the cache layer first so that it could be server to the user. than asynchronously update the db.
but there is a risk factor that if the asynchronous operation to the database fails than the db and the cache will be inconsistant.     

Scaling:

1) Vertical Scaling: Adding more resources to a single server and making it powerful so that it can handle more no of requests.
Disadvante: Single point of failure, if the single server fails than the system is dead.

2) Horizontal Scaling: adding more no of powerful servers in the system so that the request can go to any server.
Disadvantage: how to distribute the request traffic amongs the server, basically load balancer 
Data consistency

while implementing horizontal scaling we have to ensure that the servers are stateless that is it doesnot contain any information of the past interaction.

Load Balancer: Load Balancer decides to which server the request will be redirected to.
Load Balancer algorithms:

1) Round Robin: it sends the request in a circular order, first request goes to server 1 than 2 than 3 and after all servers are over it gives the request again to server 1.
this is suitable when the weight of request or the complexicity of the request is identical so that it doesnot matter to which server the request goes.

2) Weighted Round Robin: so in this case there are multiple servers and some of the server might have higher compute power , more resources.
so this algorithm sends more amount of request to the powerful servers.
but still it fails to send request as per their complexicity , it might send 3 simpler requests to the server which has more compute and 1 complex request to server which has less compute resources.

3) Least Connection: 
in this algorithm the system keeps a count of active connections of a server, means how many active requests a server is serving currently and on the basis of that it will distribute the load.

Database Scaling:
As we do horizontal scaling for servers, similar to that we have increase the database copies as well.
there is a single master node/copy of the database and then there are multiple copies which are insync with the master copy which sync the changes made in the master node in itself.
this slave nodes are only read only nodes , any read request comes , so it will be handled by this slave nodes and insert update queries will be handleled by our master node.

Database Sharding:
Database Sharding is a technique in which the a particular table is being divided into multiple logical parts and it is done on the basis of a sharding key.
for eg: there is a orders table which has multiple orders so we can divide them on the basis of date.
we can divide the table in 3 parts basically in 3 quarters. or in 12 months etc.

<--------------------------------------------------------------------------------->Lecture 9: Concurrency & Parallelism<---------------------------------------------------------------------------------------------------->

Concurrency:
So when we make a api call, it is a synchronous process the cpu is ideal untill it get's the response and it is a lot of time waste, cpu could have introduced many tasks in that period.
So to solve this issue we have introduced Concurrency so that the cpu doesnot remains ideal in that time and it can be alloted to some other tasks.
So Concurrency is about structuring the programme in a way that it can start, stop and continue execution of a task whenever it wants.
Concurrency can be achived by a single core cpu only.

Parallelism: 
Parallelism is also similar to concurrency in which as well it executes task simultaniously, but the difference is at a hardware level it has multiple cpu core support.
so that it can distribute the tasks between multiple cores.

Parallelism is about doing multiple things at once and Concurrency is about dealing with multiple things at once.

Io Bound vs CPU Bound:

1) Io Bound is making request and waiting for response factor.
Api call, Database interaction, interaction with file system etc all come under I/O operation and this requeires Concurrency which means when the cpu is ideal, assign it to another task.

Cpu Bound is actual computing, like processing, validating jwt token etc.

How Cpu achives Concurrency/Parallelism:

1) Threads:
Thread are light weight parts of process which run simultaniously.
Thread are allocated with memory which is stack, where the function calls, the variables declared in them are stored.
Then there are schedulers which are responsible for starting pausing and continuing the execution of threads.

2) Event Loops:
Event loops works on a single thread.
When there is a I/O task than the event loop delegates(gives) it to someone else and when the execution is complete it adds the task in a callback queue, and after that eventloop get's the callback 
from callback queue and executes the callback.
